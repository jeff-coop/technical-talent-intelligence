# GitHub Signals for Recruiters

This page documents how I evaluate GitHub profiles as part of understanding technical talent.

GitHub should never be used as a pass/fail hiring signal.

Many exceptional engineers have little or no public GitHub activity.

However, when GitHub activity is available, it can provide useful context about technical interests, learning habits, ownership, and community involvement.

The goal is not to judge candidates.

The goal is to better understand how engineers spend time outside of their day-to-day responsibilities.

---

# First Thing I Look At

## Pinned Repositories

Pinned repositories often provide the fastest insight into a candidate's interests.

Questions:

* What did they choose to showcase?
* Are these personal projects?
* Open source contributions?
* Research projects?
* Tools they built?

Examples:

* Infrastructure Projects
* AI Projects
* Developer Tools
* Research Work
* Data Projects

Strong candidates often use pinned repositories intentionally.

---

# Repository Quality

I care less about quantity and more about quality.

Things I look for:

* Clear README
* Documentation
* Project Structure
* Thoughtful Design
* Evidence of Ownership

A single well-built project can be more impressive than dozens of unfinished projects.

---

# Contribution Activity

Contribution activity can provide context.

Questions:

* Are they consistently active?
* Do they contribute over long periods?
* Are they maintaining projects?

Consistency often matters more than intensity.

---

# Open Source Contributions

One of the strongest signals when present.

Examples:

* Pull Requests
* Bug Fixes
* Documentation Improvements
* Feature Development
* Community Participation

Contributing to existing projects often demonstrates:

* Collaboration
* Communication
* Technical Curiosity

---

# Pull Requests

Things I pay attention to:

* Complexity
* Discussion Quality
* Code Review Participation
* Technical Communication

Pull requests often reveal how engineers collaborate.

---

# Stars

Stars alone are usually weak signals.

However, they can provide context.

Questions:

* What projects do they follow?
* What ecosystems interest them?
* What technologies are they learning?

Examples:

* Kubernetes
* PostgreSQL
* PyTorch
* Kafka
* ClickHouse

Sometimes stars reveal technical interests before they appear on a resume.

---

# Forks

Forks are often misunderstood.

A fork by itself means very little.

More interesting questions:

* Did they contribute back?
* Did they build something from the fork?
* Was it used for learning?

---

# Commit Activity

Commit activity should be interpreted carefully.

Not all engineering work happens publicly.

Not all companies allow open-source contributions.

I generally avoid drawing strong conclusions from commit counts alone.

---

# Weekend Activity

I do not use weekend activity as a hiring signal.

People have different priorities, responsibilities, and schedules.

However, personal projects, open source, and learning outside of work can sometimes demonstrate curiosity and passion for a topic.

The context matters.

---

# Technical Writing

One of the most underrated signals.

Examples:

* README Files
* Documentation
* Technical Blogs
* Project Explanations

Strong communication frequently appears among strong engineers.

---

# Breadth vs Depth

## Breadth

Many projects across multiple areas.

May indicate:

* Curiosity
* Exploration
* Learning

## Depth

Long-term investment in a project or ecosystem.

May indicate:

* Ownership
* Commitment
* Technical Depth

Both can be valuable.

---

# Signals I Find Most Interesting

Not ranked.

Examples:

* Open Source Contributions
* Maintainer Experience
* Long-Term Projects
* Technical Writing
* Research Projects
* Developer Tools
* Infrastructure Projects
* AI Projects
* Teaching Projects
* Community Involvement

---

# Signals I Care Less About

Examples:

* Raw Commit Counts
* Number of Repositories
* Number of Followers
* Number of Stars Received

These metrics often lack context.

---

# Questions I'm Trying To Answer

When reviewing a GitHub profile, I am usually trying to understand:

* What interests this person?
* What do they learn outside of work?
* What do they build?
* How do they communicate?
* How do they collaborate?
* What technical communities do they participate in?

The goal is not to evaluate code quality.

The goal is to better understand the person behind the profile.
